# 開発環境セットアップ（どのPC・場所でも同じ手順）

このドキュメントでは、**別のPC**や**別の場所**（自宅・外出先・別オフィスなど）で、同じ開発環境を短時間で構築する手順をまとめています。

---

## 新しいPCで OneDrive のフォルダを開いたとき（最短）

**他のPCで開発していたフォルダを OneDrive に入れていて、新しいPCでそのフォルダを開いた場合**は、特別な設定は不要です。次の **1 コマンド** を最初に 1 回だけ実行してください。

```bash
npm run setup
```

これで以下が自動で行われます。

- `.env.local` が無ければ `.env.example` からコピー（あればそのまま利用）
- 別PCで作られた `.next` を削除（このPC用にクリーンな状態に）
- `npm install` でこのPC用の依存関係をインストール

あとは **`npm run dev`** で開発を始められます。  
初めてこのPCで `.env.local` を作成した場合は、中身（Supabase のURL・キーなど）を編集してから `npm run dev` を実行してください。既に OneDrive で `.env.local` が同期されていれば、そのまま `npm run dev` で動きます。

---

## 前提条件（どの環境でも共通）

| 項目 | 要件 |
|------|------|
| **Node.js** | **20.9 以上**（Next.js 16 の要件） |
| **npm** | Node に同梱（通常はそのままで可） |
| **Git** | リポジトリを clone する場合に必要 |

### Node のバージョン確認

```bash
node -v
```

`v20.9.0` 以上であることを確認してください。不足している場合は [Node.js 公式](https://nodejs.org/) から LTS（20.x）をインストールするか、下記「Node のバージョン管理（任意）」を参照してください。

---

## 手順 1：プロジェクトを用意する

### 方法 A：Git で clone する（推奨）

```bash
git clone <リポジトリのURL> 2026clHP
cd 2026clHP
```

### 方法 B：OneDrive / クラウド同期フォルダで開く

- 既にプロジェクトが同期フォルダにある場合は、そのフォルダを開くだけです。
- **新しいPCで開いたら、まず `npm run setup` を 1 回だけ実行**すれば、依存関係のインストールと .env.local の準備がまとめて行われ、同じ環境で開発できます（上記「新しいPCで OneDrive のフォルダを開いたとき」参照）。

---

## 手順 2：Node のバージョンを合わせる（任意だが推奨）

プロジェクトルートに `.nvmrc` があります。**nvm**（Node Version Manager）を使っている場合：

```bash
nvm use
```

または、手動で Node 20 をインストール・切り替えしたうえで、`node -v` が 20.9 以上であることを確認してください。

---

## 手順 3：依存関係のインストール

プロジェクトルートで実行します。

```bash
npm install
```

- 初回や、別PCにプロジェクトをコピーした直後は**必ず実行**してください。
- OneDrive 等で `node_modules` を同期していない場合も、各PCでこのコマンドを実行する必要があります。

---

## 手順 4：環境変数の設定

1. プロジェクトルートに `.env.local` を作成します。
2. 中身は `.env.example` をコピーして、実際の値に書き換えます。

```bash
# Windows（PowerShell）
Copy-Item .env.example .env.local
# その後 .env.local をエディタで開いて値を編集

# Mac / Linux
cp .env.example .env.local
# その後 .env.local をエディタで開いて値を編集
```

### 最低限必要な環境変数（開発のみでよい場合）

| 変数名 | 説明 | 取得元 |
|--------|------|--------|
| `NEXT_PUBLIC_SUPABASE_URL` | Supabase プロジェクトURL | Supabase ダッシュボード → Project Settings → API |
| `NEXT_PUBLIC_SUPABASE_ANON_KEY` | Supabase 匿名キー（公開用） | 同上 |
| `SUPABASE_SERVICE_ROLE_KEY` | Supabase サービスロールキー（管理用・秘密） | 同上（service_role） |

- メール送信・Stripe・本番URL などは、必要に応じて `.env.local` に追加してください。詳細は `.env.example` のコメントを参照してください。
- **重要**：`.env.local` は Git にコミットしないでください（`.gitignore` で除外済みです）。

---

## 手順 5：開発サーバーの起動

```bash
npm run dev
```

- ブラウザで **http://localhost:3000** を開くと表示されます。
- **同じネットワーク内の別PC**から見る場合は、開発サーバーを動かしているPCのIPアドレスを使います（例：`http://192.168.1.10:3000`）。  
  開発用に `next dev` は `-H 0.0.0.0` で起動するよう設定済みです。

### よく使うコマンド

| コマンド | 説明 |
|----------|------|
| `npm run dev` | 開発サーバー起動（ホットリロード） |
| `npm run build` | 本番用ビルド |
| `npm run start` | ビルド後の本番モード起動 |
| `npm run clean` | `.next` を削除（ビルドエラー時など） |
| `npm run build:fresh` | clean のあとビルド |

---

## OneDrive / クラウド同期フォルダで開発する場合

- **node_modules** と **.next** はファイル数が多く、同期が重くなったり不整合の原因になりやすいです。
- **推奨**：
  - 各PCで **必ず `npm install` を実行**する。
  - `.next` はビルド成果物のため、同期しなくてよい（毎回 `npm run dev` で再生成される）。
- 可能であれば、`.gitignore` に含まれる `node_modules` と `.next` は同期対象外にするか、新規PCでは clone 後に `npm install` で一から入れる運用にすると安全です。

---

## OS 別の補足

- **Windows**  
  - コマンドは PowerShell または「Node.js コマンドプロンプト」で実行できます。  
  - パスに日本語が含まれる場合、問題が出たらプロジェクトを英数字のみのパスに置くか、WSL で試してください。

- **Mac / Linux**  
  - 上記手順のコマンドはそのまま利用できます。  
  - Node の導入は `nvm` や `fnm` を使うと便利です。

- **どのOSでも**  
  - 同じ `package.json` と `.nvmrc` を使うことで、**同じ Node バージョン・同じ依存関係**で開発できます。

## トラブルシューティング

| 現象 | 対処 |
|------|------|
| `node -v` が 20.9 未満 | Node.js 20 LTS をインストールするか、`nvm use` で 20 に切り替える。 |
| `npm install` でエラー | ネット接続を確認。`npm cache clean --force` のあと再度 `npm install`。 |
| `npm run dev` でポート使用エラー | 3000 番が使用中なら、別ターミナルで使用プロセスを終了するか、`next dev -p 3001` のように別ポートで起動。 |
| 別PCからアクセスできない | 開発サーバーを起動しているPCのファイアウォールでポート 3000 を許可する。同一LANか確認。 |
| ビルドエラーが続く | `npm run build:fresh` でクリーンビルドを試す。 |
| 認証・Supabase のエラー | `.env.local` に `NEXT_PUBLIC_SUPABASE_URL` と `NEXT_PUBLIC_SUPABASE_ANON_KEY` が正しく設定されているか確認し、`npm run dev` を再起動。 |
