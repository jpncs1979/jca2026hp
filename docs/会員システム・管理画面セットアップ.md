# 会員マイページ・事務局管理システム セットアップ手順

## 概要

会員マイページ（`/mypage`）と事務局管理ダッシュボード（`/admin`）の構築が完了しました。

## 1. データベース（Supabase）

### マイグレーション実行

Supabase ダッシュボードの SQL エディタで、以下の順に実行してください。

1. `supabase/migrations/001_create_tables.sql`（既存）
2. `supabase/migrations/002_add_email_to_applications.sql`（既存）
3. `supabase/migrations/003_member_system.sql`（既存）
4. `supabase/migrations/004_member_import_ica.sql`（会員取り込み・ICA会員・会友対応）
5. `supabase/migrations/005_profiles_read_by_email.sql`（インポート会員がログイン時に会員証を表示できるようにする）

### 初回管理者の作成

`profiles` テーブルに `is_admin: true` のレコードが必要です。以下の手順で作成してください。

#### 手順 1: ユーザーを Supabase Auth に登録する

**方法 A: Supabase ダッシュボードから登録**

1. Supabase ダッシュボード → **Authentication** → **Users**
2. **Add user** → **Create new user** をクリック
3. メールアドレスとパスワードを入力（例: `admin@jp-clarinet.org`）
4. **Create user** をクリック

※「Auto Confirm User」にチェックを入れると、メール確認なしで即ログイン可能です。

**方法 B: サイトのマイページから登録（サインアップ機能がある場合）**

1. `/mypage` にアクセス
2. サインアップフォームでメールアドレスとパスワードを登録

#### 手順 2: プロフィールが自動作成されているか確認する

`handle_new_user` トリガーにより、Auth にユーザーが作成されると自動で `profiles` にレコードが追加されます。

Supabase ダッシュボード → **Table Editor** → **profiles** で、登録したメールアドレスの行が存在することを確認してください。

※プロフィールが無い場合は、以下で手動作成できます（`auth.users` の `id` を確認してから実行）:

```sql
-- auth.users の id を確認
SELECT id, email FROM auth.users WHERE email = 'admin@jp-clarinet.org';

-- プロフィールを手動作成（上記で取得した id を user_id に指定）
INSERT INTO profiles (user_id, name, name_kana, email, status, is_admin)
VALUES (
  'ここにauth.usersのidを貼り付け',
  '管理者',
  'かんりしゃ',
  'admin@jp-clarinet.org',
  'active',
  true
);
```

#### 手順 3: 既存プロフィールを管理者に昇格する

プロフィールが既にある場合、SQL で `is_admin` を更新します。

```sql
-- メールアドレスで指定して管理者に昇格
UPDATE profiles
SET is_admin = true, status = 'active'
WHERE email = 'admin@jp-clarinet.org';
```

#### 手順 4: ログインして動作確認する

1. `/mypage` にアクセスし、手順 1 で登録したメールアドレスとパスワードでログイン
2. ログイン後、`/admin` にアクセス
3. ダッシュボードや会員管理が表示されれば成功です

---

### ダッシュボード（/admin）に入れない場合

ログインはできるが `/admin` にアクセスするとマイページに戻される場合、以下を確認してください。

**1. SUPABASE_SERVICE_ROLE_KEY を設定する（最重要）**

`.env.local` に以下が設定されていないと、管理者チェックが動作しません。

```env
SUPABASE_SERVICE_ROLE_KEY=eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...（長い文字列）
```

**取得方法**: Supabase ダッシュボード → **Project Settings** → **API** → **Project API keys** の **service_role**（secret）をコピー

※このキーはサーバーサイドでのみ使用します。絶対にクライアントに公開しないでください。

**2. 設定後は開発サーバーを再起動**

```bash
# Ctrl+C で停止後
npm run dev
```

**3. profiles で is_admin を確認**

Supabase → **Table Editor** → **profiles** で、あなたのメールアドレスの行の **is_admin** が `true` になっているか確認してください。`false` の場合は、上記「管理者に昇格する SQL」を実行してください。

---

### ログインできない場合の対処

**「メールアドレスまたはパスワードが正しくありません」と表示される場合**

→ そのメールアドレスのユーザーが Supabase Auth に存在しないか、パスワードが違います。以下を順に確認してください。

**管理者アカウント（事務局）で入れない場合**

1. **Supabase にユーザーがいるか確認**  
   Supabase ダッシュボード → **Authentication** → **Users** で、ログインに使っているメールアドレス（例: `jpncs1979@gmail.com`）の行があるか確認します。
2. **ユーザーがいない場合**  
   下記「Supabase ダッシュボードでユーザーを追加する」の手順で作成し、続けて「管理者に昇格する SQL」を実行してください。
3. **ユーザーはいるがパスワードがわからない／違う場合**
   - **別の管理者がいる場合**: その方に、会員一覧であなたの会員プロフィールを開き「パスワード再設定メールを送信」してもらってください。届いたメールのリンクから新しいパスワードを設定できます。
   - **管理者が自分だけの場合**: Supabase ダッシュボード → **Authentication** → **Users** で該当ユーザーを開き、「Send password recovery email」があれば利用するか、やむを得ない場合は一度ユーザーを削除し、同じメールで「Add user」から作り直し、新しいパスワードを設定してください。作り直したあと、`profiles` の `user_id` がずれる場合は、同じメールの行の `user_id` を auth.users の新しい id に合わせて更新してください。
4. **管理者として認識されるようにする**  
   `profiles` にそのメールの行があり、**is_admin** が `true` になっているか確認します。`false` の場合は下記「管理者に昇格する SQL」を実行してください。

**Supabase ダッシュボードでユーザーを追加する（推奨）**

1. Supabase ダッシュボード → **Authentication** → **Users**
2. **Add user** → **Create new user**
3. **Email**: `jpncs1979@gmail.com`（または使用するメールアドレス）
4. **Password**: `cl11071107`（または使用するパスワード）
5. **Auto Confirm User** にチェックを入れる（メール確認をスキップ）
6. **Create user** をクリック

**管理者に昇格する SQL を実行**

```sql
-- プロフィールを管理者に昇格（メールアドレスを実際のものに変更）
UPDATE profiles
SET is_admin = true, status = 'active'
WHERE email = 'jpncs1979@gmail.com';
```

**memberships とは**

`memberships` は**会員の契約・有効期限**を管理するテーブルです。入会日・有効期限・支払方法などを保存します。

- **事務局ダッシュボード（/admin）へのアクセス**: `memberships` は不要。`is_admin = true` だけでアクセスできます。
- **マイページのデジタル会員証**: 有効期限を表示するために `memberships` のレコードが必要です。レコードが無い場合は「有効期限: —」と表示されます。

初回管理者でマイページの会員証に有効期限を表示したい場合のみ、以下を実行してください。

```sql
-- 1. プロフィールIDを確認
SELECT id, email FROM profiles WHERE email = 'jpncs1979@gmail.com';

-- 2. 上記で表示された id（UUID）を profile_id に指定して実行
INSERT INTO memberships (profile_id, join_date, expiry_date, payment_method)
VALUES (
  'xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx',  -- ここに profiles.id を貼り付け
  CURRENT_DATE,
  CURRENT_DATE + INTERVAL '1 year',
  'transfer'
);
```

**メール確認が有効な場合**

Supabase の **Authentication** → **Providers** → **Email** で「Confirm email」が ON の場合、サインアップ後は確認メールのリンクをクリックするまでログインできません。ダッシュボードで「Add user」する際は **Auto Confirm User** にチェックすれば即ログイン可能です。

**Supabase Auth の URL 設定（localhost で開発する場合に重要）**

ログインがうまくいかない場合、Supabase の認証 URL 設定が原因のことがあります。

1. Supabase ダッシュボード → **Authentication** → **URL Configuration**
2. **Site URL**: ローカル開発の場合は `http://localhost:3000` を設定
3. **Redirect URLs**: `http://localhost:3000/**` を追加（複数ある場合は改行で区切る）
4. **Save** をクリック

※ 本番環境では、Site URL と Redirect URLs に本番ドメイン（`https://jca2026hp.vercel.app/**`）を設定してください。設定後、開発サーバーを再起動してください。

**パスワード再設定メールのリンクをクリックしてもマイページに飛ばない場合**

事務局が送る「パスワード再設定メール」のリンクは、Supabase で検証後に **Redirect URLs** に登録された URL へリダイレクトされます。

1. **Redirect URLs** に次のどちらか（または両方）が含まれているか確認してください。
   - 開発: `http://localhost:3000/auth/callback` または `http://localhost:3000/**`
   - 本番: `https://jca2026hp.vercel.app/auth/callback` または `https://jca2026hp.vercel.app/**`
2. **NEXT_PUBLIC_SITE_URL**（`.env.local`）を、実際にアクセスするサイトの URL に設定してください。
   - 開発: `http://localhost:3000`（末尾のスラッシュなしでも可）
   - 本番: `https://jca2026hp.vercel.app`
   - 未設定の場合は `http://localhost:3000` が使われます。

リンククリック後はいったん `/auth/callback` に飛び、そこでトークン処理をしてから **パスワード再設定専用ページ**（`/auth/set-password`）に転送されます。新しいパスワードを設定するとマイページへ移動します。

**メールのリンクをクリックした先が再設定ページにならない場合**

1. **Redirect URLs の確認**  
   Supabase の **Redirect URLs** に、開発なら `http://localhost:3000/**`、本番なら `https://jca2026hp.vercel.app/**` が含まれているか確認してください。含まれていないと、Supabase が別の URL（Site URL のみ）へリダイレクトし、再設定ページに届かないことがあります。
2. **本番の場合は NEXT_PUBLIC_SITE_URL**  
   Vercel などにデプロイしている場合、Vercel の環境変数で **NEXT_PUBLIC_SITE_URL** を本番の URL（`https://jca2026hp.vercel.app`）に設定してください。設定が違うとメールのリンク先が誤ります。
3. **フォールバック**  
   サイト側で、メールのリンクから「トップページやその他のページ」に飛んだ場合でも、自動でパスワード再設定ページへ転送する処理を入れています。リンクをクリックしたあと、どこかしらサイト内のページが開いていれば、すぐに再設定ページに切り替わる想定です。それでも再設定ページにならない場合は、上記 1・2 を確認し、**新しいパスワード再設定メールを送り直して**、届いたリンクを再度開いてみてください。

**パスワード再設定メール送信で「権限がありません」と表示される場合**

→ 事務局用アカウントでログインし直してください。同じブラウザで会員がパスワード再設定をした直後は、その会員のログイン状態になっていることがあります。一度ログアウトしてから、事務局用のメール・パスワードで会員マイページにログインし、再度「事務局管理」から会員一覧を開いて送信してください。それでも表示される場合は、Supabase の **Table Editor** → **profiles** で、ご自身のメールアドレスの行の **is_admin** が `true` か確認してください。

**「認証の準備ができていません」と表示される場合**

→ `.env.local` に **NEXT_PUBLIC_SUPABASE_URL** と **NEXT_PUBLIC_SUPABASE_ANON_KEY** が設定されていません。Supabase ダッシュボード → **Project Settings** → **API** から取得し、設定後に開発サーバーを再起動してください。

## 2. 環境変数

### 重要：事務局ダッシュボードに入れない場合

**SUPABASE_SERVICE_ROLE_KEY が未設定**だと、ログインしていても `/admin` にアクセスできません。必ず設定してください。

1. Supabase ダッシュボード → **Project Settings** → **API**
2. **Project API keys** の **service_role**（secret）をコピー
3. `.env.local` に `SUPABASE_SERVICE_ROLE_KEY=コピーしたキー` を追加
4. 開発サーバーを再起動（`npm run dev`）

---

`.env.local` に以下を追加してください。

```env
# Supabase（既存）
NEXT_PUBLIC_SUPABASE_URL=
NEXT_PUBLIC_SUPABASE_ANON_KEY=

# 事務局管理用（必須）
SUPABASE_SERVICE_ROLE_KEY=

# Stripe 決済
STRIPE_SECRET_KEY=
STRIPE_WEBHOOK_SECRET=
# パスワード再設定メールのリダイレクト先に使用。開発は http://localhost:3000、本番は https://jca2026hp.vercel.app
NEXT_PUBLIC_SITE_URL=https://jca2026hp.vercel.app

# メール送信（入会承認メール用・既存）
GMAIL_USER=
GMAIL_APP_PASSWORD=
```

## 3. Stripe 設定

### 3-1. API キーを取得する

1. [Stripe ダッシュボード](https://dashboard.stripe.com) にログイン
2. 右上の「開発者」→「API キー」を開く
3. 「シークレットキー」の「公開」をクリックして表示
4. 表示されたキー（`sk_test_...` または `sk_live_...`）をコピー
5. `.env.local` の `STRIPE_SECRET_KEY=` の後に貼り付けて保存
   - テスト時は `sk_test_...`、本番は `sk_live_...` を使用

### 3-2. Webhook エンドポイントを追加する

**Webhook とは**：決済が完了したとき、Stripe が自動でサイトの URL に通知を送る仕組みです。これにより申込の支払い状況が自動更新されます。

1. Stripe ダッシュボードで「開発者」→「Webhook」を開く
2. 「エンドポイントを追加」をクリック
3. 「エンドポイント URL」に以下を入力：
   ```
   https://jca2026hp.vercel.app/api/webhooks/stripe
   ```
   - 独自ドメインを使う場合は、そのドメインに読み替えてください。

### 3-3. イベントを選択する

1. 「イベントを選択」をクリック
2. 検索欄に `checkout.session.completed` と入力
3. 一覧から **`checkout.session.completed`** を選択して追加
4. 「エンドポイントを追加」をクリック

※ このイベントは「チェックアウト（決済）が完了したとき」に通知が送られます。

### 3-4. Webhook 署名シークレットを設定する

1. 追加した Webhook の詳細画面を開く
2. 「署名シークレット」の「表示」をクリック
3. 表示された文字列（`whsec_...`）をコピー
4. `.env.local` の `STRIPE_WEBHOOK_SECRET=` の後に貼り付けて保存
5. 開発サーバーを再起動（`npm run dev` を一度止めて再度実行）

### 3-5. ローカル開発の場合

本番 URL がない場合は Stripe CLI でローカル用の Webhook を用意します。

1. [Stripe CLI](https://stripe.com/docs/stripe-cli) をインストール
2. ターミナルで以下を実行：
   ```bash
   stripe listen --forward-to localhost:3000/api/webhooks/stripe
   ```
3. 表示される `whsec_...` を `.env.local` の `STRIPE_WEBHOOK_SECRET` に設定

## 4. 主な機能

### 会員マイページ（`/mypage`）

- ログイン（Supabase Auth）
- デジタル会員証
- プロフィール編集（住所・連絡先・所属）
- 申込履歴（ヤングコンクール等）
- 会員限定コンテンツ（会報PDF・動画リンク）

### 事務局管理（`/admin`）

- **会員管理**:
  - 承認待ち一覧、承認ボタン（入会承認メール送信）
  - **会員のパスワード再設定**: 会員一覧で対象の会員をクリックして会員詳細を開き、「**パスワード再設定メール**」の欄にある「パスワード再設定メールを送信」ボタンから、その会員に再設定用メールを送信できます。会員本人がメールのリンクを開いて新しいパスワードを設定します。
  - Excel 取り込み（会員データ一括インポート）
  - ICA会員のみ・会員種別（正会員/学生会員/賛助会員/会友）・未納者のみ でフィルター
  - 条件指定でメール一括送信（未納者への催促など）
  - 一括有効期限延長・CSS 登録者への会員資格付与
  - CSV エクスポート
- **コンクール管理**: 第15回ヤングコンクール申込者一覧、動画URLのワンクリック視聴・コピー

### 申込フロー（ヤングコンクール）

- 銀行振込 or クレジットカード（Stripe）を選択可能
- 同時入会: 参加料＋初年度会費の合算で決済
- 決済成功時に `payment_status` を自動更新

## 5. 会員データ Excel 取り込み

会員管理画面の「Excel取り込み」から、既存の会員データを一括登録できます。

### 必要な列

| Excel 列名 | 必須 | 説明 |
|-----------|------|------|
| 名前 | ○ | 氏名 |
| 名前(カナ) | - | ふりがな |
| システム用メールアドレス | ○ | メールアドレス（重複時は更新） |
| 会員種別 | - | 正会員、学生会員、賛助会員、会友 など |
| 会員有効終了日 | - | 有効期限（例: 2027/01/31） |
| ICA資格 | - | 「会員」= ICA会員 |
| 会費支払い方法 | - | CSS、振込 など |
| 電話番号、住所_郵便番号、住所_都道府県… | - | 任意 |

### メール送信（条件指定・選択送信）

会員管理の「メール送信」より、条件指定または選択した会員へ一括メール送信ができます。

**送信制限**

- **1回あたり**: 最大 100 件まで（Gmail のレート制限対策）
- **Gmail 個人アカウント**: 1日 500 通まで
- **Google Workspace**: 1日 2,000 通程度（プランによる）

100 件を超える場合は、条件を絞るか、複数回に分けて送信してください。

`.env.local` に `EMAIL_USER` と `EMAIL_APP_PASSWORD`（Gmail アプリパスワード）を設定してください。

## 7. 本番用ビルド（npm run build）でエラーが出るとき

「操作が許可されていません」（EPERM）や、ビルドが途中で止まるときは、**ビルド用の一時フォルダ（.next）がロックされている**ことが原因です。次の手順で対処してください。

### 手順（どちらか一方でOK）

**方法A：コマンドでまとめてやる（おすすめ）**

1. **開発サーバーを止める**  
   ターミナルで `npm run dev` を実行している場合は、そのターミナルで **Ctrl + C** を押して止めます。
2. **ターミナルを開く**  
   プロジェクトのフォルダ（`2026clHP` など）を開いた状態で、ターミナル（PowerShell やコマンドプロンプト）を開きます。
3. **次のコマンドを1つ入力して Enter**  
   ```  
   npm run build:fresh  
   ```  
   これで「.next フォルダを削除 → ビルド」が自動で実行されます。  
   「EPERM」「ロックされています」と出た場合は、方法Bの「1. 開発サーバーを止める」をもう一度確認し、しばらく待ってから再度 `npm run build:fresh` を試してください。

**方法B：フォルダを手動で消してからビルドする**

1. **開発サーバーを止める**  
   ターミナルで `npm run dev` を実行している場合は、**Ctrl + C** で止めます。
2. **.next フォルダを削除する**  
   - エクスプローラーでプロジェクトのフォルダ（例：`2026clHP`）を開く  
   - 中にある **`.next`** という名前のフォルダを探す（先頭がピリオドのフォルダです）  
   - その **`.next`** フォルダを右クリック → **削除** を選ぶ  
   - 削除できない場合は、開発サーバーを止めたあと、しばらく待ってからもう一度試す。OneDrive で同期しているフォルダの場合は、同期が落ち着くまで待つと消せることがあります。
3. **ビルドを実行する**  
   ターミナルで次のコマンドを入力して Enter を押します。  
   ```  
   npm run build  
   ```

### 用語の説明

- **ビルド**：本番用のサイトを作る処理です。Vercel などにデプロイするときも、自動でこの処理が行われます。
- **.next フォルダ**：ビルドや開発サーバーが作る「一時的な成果物」が入るフォルダです。削除しても、次に `npm run dev` や `npm run build` を実行すれば自動で再作成されます。

## 8. 注意事項

- 管理者は `profiles.is_admin = true` のユーザーのみアクセス可能
- サインアップ時、`auth.users` に INSERT されると `profiles` が自動作成（`handle_new_user` トリガー）
- 初回管理者は SQL で `is_admin` を更新する必要があります
